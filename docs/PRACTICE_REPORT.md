## Введение

Во время практики я самостоятельно разработал учебный стенд для изучения Python. В моей системе студент решает задачи в Moodle, CodeRunner проверяет программу через изолированный Jobe Server, а собственный Moodle-плагин показывает разбор решения от локальной языковой модели.

Обычная автоматическая проверка точно сообщает, прошёл ли код тесты, но начинающему студенту часто не хватает объяснения причины ошибки. Поэтому я дополнил CodeRunner безопасной обучающей подсказкой. При этом итог проверки по-прежнему определяет CodeRunner, а ИИ не получает скрытые тесты и не выдаёт готовое решение по умолчанию.

## Цель и задачи

Цель моей практики — создать стенд на Moodle 5.2.1, который можно заново развернуть и использовать для проверки программ и разбора решений.

Для достижения цели я поставил следующие задачи:

1. подобрать совместимые версии Moodle, PHP, MariaDB, CodeRunner и Jobe;
2. описать все сервисы в Docker Compose;
3. автоматизировать первую установку Moodle и плагинов;
4. настроить реальное выполнение Python через Jobe;
5. сделать небольшой сервис анализа кода на FastAPI;
6. добавить статический анализ Python через `ast` без выполнения кода;
7. разработать собственный плагин `local_aicodehelper`;
8. добавить кнопку анализа в попытку CodeRunner стандартными средствами Moodle;
9. исключить передачу скрытых тестов в сервис анализа;
10. подготовить курс по Python и автоматические проверки;
11. написать инструкцию переноса решения на действующий Moodle МИФИ.

## Исходные требования

По условию задания я использовал Moodle строго версии 5.2.1. Я не изменял ядро Moodle, исходники CodeRunner, Jobe и сжатые JavaScript-файлы сторонних плагинов. Все настройки запуска я оформил в Dockerfile, `compose.yaml` и скриптах.

Я не стал усложнять проект: не добавлял Kubernetes, Redis, очередь задач, Nginx, мониторинг и другие компоненты, которые не нужны для учебного стенда.

Основные требования к результату:

- запуск после клонирования репозитория;
- сохранение базы и файлов Moodle после перезапуска;
- выполнение Python только через Jobe;
- локальная модель без внешнего облачного API;
- отправка данных из Moodle в сервис анализа;
- защита скрытых тестов;
- запасной статический анализ при недоступной модели;
- возможность переноса собственного плагина на другой Moodle 5.2.1.

## Как работает стенд

```text
Браузер студента
       |
       v
Moodle + мой плагин ----------> MariaDB
       |
       +-- CodeRunner ----------> Jobe
       |
       +-- запрос на анализ ----> сервис анализа ---> Ollama
```

Я разделил точную проверку и обучающий разбор. Пользовательский код выполняется только в Jobe. Мой сервис читает структуру Python-кода через модуль `ast` и обращается к Ollama, но не запускает программу. Браузер работает только с Moodle и не видит внутренний адрес сервиса анализа.

## Ход работы

### Настройка Docker

Я описал в `compose.yaml` пять основных частей стенда: `moodle`, `db`, `jobe`, `ai-service` и `ollama`. Наружу я открыл только порт Moodle. Для базы, файлов Moodle и моделей Ollama я создал постоянные хранилища Docker.

Для внутренней Docker-сети я использовал тестовую подсеть `198.18.0.0/24`. Она не публикует сервисы в локальную сеть, а контейнеры всё равно обращаются друг к другу по понятным именам `db`, `jobe`, `ai-service` и `ollama`, а не по IP-адресам.

Для каждой основной части я добавил простую проверку готовности. Благодаря этому Moodle начинает установку только после запуска базы, Jobe и сервиса анализа.

### Автоматическая установка Moodle

В `moodle/Dockerfile` я закрепил Moodle 5.2.1, PHP 8.3.28, CodeRunner 5.9.2 и обязательное дополнение версии 1.4.5. Плагины копируются во время сборки образа, поэтому после запуска контейнера не нужны ручные исправления.

В `moodle/entrypoint.sh` я реализовал следующую последовательность:

1. проверка обязательных переменных окружения;
2. ожидание MariaDB;
3. создание `config.php` при первом старте;
4. установка Moodle только в пустую базу;
5. выполнение `upgrade.php`;
6. настройка адресов Jobe и сервиса анализа;
7. установка демонстрационного курса;
8. очистка кэша Moodle.

При повторном запуске скрипт использует существующую базу и не переустанавливает Moodle.

### CodeRunner и Jobe

Я настроил CodeRunner на обращение к сервису `jobe` по имени внутри Docker-сети. Для проверки я выполнил через Jobe программу:

```python
print("Hello from Jobe")
```

Jobe вернул:

```text
Hello from Jobe
```

Таким образом я проверил не только доступность порта, но и настоящее выполнение Python-кода.

### Сервис анализа кода

Я создал небольшой сервис на FastAPI с двумя адресами:

- `GET /health` — проверка работы сервиса и название выбранной модели;
- `POST /api/v1/analyze` — разбор кода и результата CodeRunner.

До обращения к Ollama я разбираю Python-код стандартным модулем `ast`. Статический анализ определяет синтаксическую ошибку, наличие циклов и функций, примерную сложность, опасные `eval` или `exec` и признаки возможного хардкода.

Я намеренно не запускаю пользовательский код внутри сервиса анализа. Также я ограничил размер кода, время ожидания ответа и не записываю полный код в журнал работы.

Если Ollama недоступна или возвращает ответ неправильного формата, я показываю результат обычного статического анализа. Благодаря этому Moodle продолжает работать даже при сбое модели.

### Разработка собственного Moodle-плагина

Я разместил собственный плагин в `moodle/local_aicodehelper`. Для связи с CodeRunner я использовал стандартный механизм событий Moodle 5.2.1 и не изменял сторонние плагины.

После проверки решения мой плагин добавляет кнопку «Проанализировать решение с помощью ИИ». По нажатию сервер Moodle сам собирает условие, название вопроса, код, оценку, статус и разрешённые результаты тестов.

При обработке нажатия кнопки я проверяю:

- авторизацию пользователя;
- `sesskey`;
- право доступа `local/aicodehelper:analyzeattempt`;
- принадлежность или доступ к попытке;
- лимит анализов;
- наличие результата CodeRunner.

Ответ модели я вывожу через отдельный класс. Весь текст очищается, поэтому модель не может вставить на страницу посторонний HTML-код. Результат сохраняется, и повторное нажатие для того же решения не создаёт несколько одинаковых обращений к модели.

### Защита скрытых тестов

Очистку данных перед отправкой я вынес в отдельную часть плагина. Для скрытого теста мой плагин передаёт только его номер, признак успеха и общий тип ошибки.

Я не передаю:

- входные данные скрытого теста;
- ожидаемый результат;
- фактический закрытый вывод;
- test code;
- дополнительные данные шаблона;
- эталонное решение преподавателя.

Я добавил отдельные проверки с метками `SECRET_`. Проверка завершается ошибкой, если хотя бы одна такая метка попадает в данные, отправляемые сервису анализа.

### Учебный курс Python

Я подготовил курс «Основы Python: практика с CodeRunner». Курс содержит пять тем, шесть страниц с теорией и пятнадцать программных задач.

В курс вошли:

1. ввод, переменные и арифметика;
2. условия;
3. циклы;
4. списки;
5. строки и словари.

Для каждой задачи я добавил два открытых примера и один скрытый тест. Вопросы используют обычное текстовое поле CodeRunner, поэтому курс не зависит от редактора Ace. Установщик создаёт курс стандартными средствами Moodle и не создаёт копию при повторном запуске.

### Подготовка переноса в Moodle МИФИ

Я отдельно описал полный лабораторный стенд и установку в существующий Moodle. Для рабочего сервера не требуется переносить лабораторную базу, `moodledata`, `config.php` и контейнер Jobe.

На действующий Moodle 5.2.1 нужно перенести только каталог моего плагина, выполнить стандартное обновление Moodle, запустить сервис анализа с Ollama и указать его внутренний адрес в настройках. Если в институте уже работает Jobe на порту `9000`, его можно оставить без изменений.

Подробную последовательность я описал в `docs/DEPLOYMENT.md`.

## Возникшие сложности и решения

Во время работы я столкнулся с несколькими практическими проблемами.

Первая проблема была связана со строгим требованием Moodle 5.2.1 и новой структурой каталогов. Я отдельно учёл публичный каталог `public` для плагинов и каталог `admin/cli` для служебных команд.

Вторая проблема возникла с редактором Ace в CodeRunner. Я не стал изменять JavaScript CodeRunner, а выбрал для учебного курса обычное текстовое поле с `uiplugin=none`. Выполнение заданий при этом осталось стабильным.

Третья проблема — нестабильный ответ небольшой модели. Я проверяю все поля ответа, а отсутствующие данные заменяю результатами обычного анализа кода. В результате ошибка Ollama не ломает страницу Moodle.

Четвёртая проблема — защита скрытых тестов. Я сделал отдельную функцию очистки отправляемых данных и проверил её несколькими автоматическими тестами.
