## Введение

Во время практики был разработан учебный стенд для обучения программированию на Python. Студент решает задачи внутри Moodle, CodeRunner проверяет программу на тестах через изолированный сервер Jobe, а собственный плагин Moodle позволяет запросить разбор ошибки у локальной языковой модели.

Актуальность работы связана с тем, что обычная автоматическая проверка сообщает результат тестов, но не всегда помогает начинающему студенту понять причину ошибки. Разработанная система дополняет точную проверку CodeRunner направляющей подсказкой, не раскрывая скрытые тесты и готовое решение.

## Цель и задачи

Цель практики — создать воспроизводимый учебный стенд на Moodle 5.2.1 для проверки программ и безопасного ИИ-анализа решений.

Для достижения цели были поставлены задачи:

1. подобрать совместимые версии Moodle, PHP, MariaDB, CodeRunner и Jobe;
2. описать запуск сервисов в Docker Compose;
3. автоматизировать первую установку Moodle и плагинов;
4. создать FastAPI-сервис статического и модельного анализа Python-кода;
5. разработать Moodle-плагин `local_aicodehelper`;
6. встроить кнопку анализа в попытку CodeRunner без изменения стороннего кода;
7. исключить передачу скрытых тестов в AI service;
8. подготовить практический курс по Python;
9. добавить автоматические и интеграционные проверки;
10. подготовить инструкцию переноса на учебный сервер.

## Требования к системе

Стенд должен запускаться на Ubuntu с Docker и Docker Compose Plugin. Наружу публикуется только Moodle. MariaDB, Jobe, AI service и Ollama доступны по именам сервисов во внутренней Docker-сети.

Основные функциональные требования:

- автоматическая установка Moodle при первом запуске;
- сохранение базы и файлов Moodle после перезапуска;
- реальное выполнение Python через Jobe;
- локальная работа модели без передачи кода внешнему облачному API;
- статический анализ Python без запуска кода в AI service;
- получение данных попытки на сервере Moodle;
- защита скрытых тестов;
- безопасный fallback при недоступной модели.

## Архитектура стенда

```text
Браузер студента
       |
       v
Moodle + local_aicodehelper ---> MariaDB
       |
       +-- CodeRunner ----------> Jobe
       |
       +-- серверный HTTP ------> AI service ---> Ollama
```

Пользовательский код выполняется только в Jobe. AI service строит AST для Python и отправляет текстовый запрос в Ollama, но не запускает программу. Браузер не получает внутренний адрес AI service.

## Ход работы

### Контейнерное окружение

В `compose.yaml` описаны пять основных сервисов: `moodle`, `db`, `jobe`, `ai-service` и `ollama`. Для базы, `moodledata` и моделей созданы Docker volumes. Healthcheck позволяют дождаться готовности зависимостей до запуска проверок.

Образ Moodle собирается из официального архива версии 5.2.1. CodeRunner, обязательный behaviour-плагин и собственный `local_aicodehelper` добавляются на этапе сборки. `entrypoint.sh` ждёт MariaDB, устанавливает Moodle только при пустой базе, выполняет upgrade и задаёт внутренние endpoint.

### Проверка программ

CodeRunner формирует программу и отправляет её в Jobe. Jobe запускает код в отдельном контейнере с ограничениями времени и памяти. В smoke-тесте выполняется программа:

```python
print("Hello from Jobe")
```

Ожидаемый результат — строка `Hello from Jobe`.

### AI service

AI service реализован на FastAPI. Endpoint `/api/v1/analyze` принимает язык, условие, код и безопасные результаты CodeRunner. Перед обращением к модели Python-код разбирается модулем `ast`. Сервис ограничивает размер запроса, использует timeout и не записывает полный код в лог.

Если Ollama не отвечает или возвращает некорректный JSON, сервис строит статический преподавательский ответ и устанавливает `fallback_used=true`. Поэтому ошибка модели не ломает страницу Moodle.

### Собственный Moodle-плагин

Плагин расположен в `moodle/local_aicodehelper`. Он использует hooks Moodle 5.2.1 и не меняет CodeRunner или ядро Moodle.

После проверки решения плагин добавляет кнопку «Проанализировать решение с помощью ИИ». Сервер Moodle собирает условие, код, оценку и доступные результаты тестов. AJAX-запрос требует авторизацию, capability и корректный `sesskey`. Ответ экранируется перед выводом.

Для скрытого теста передаются только номер, факт успеха и безопасный тип ошибки. Вход, ожидаемый вывод, test code, шаблон и эталонное решение не включаются в payload.

### Учебный курс

В проект добавлен курс «Основы Python: практика с CodeRunner». Он состоит из пяти тем, шести страниц и пятнадцати программных задач. Темы охватывают ввод, условия, циклы, списки, строки и простые алгоритмы.

Каждая задача имеет два открытых примера и один скрытый тест. Эталонные решения автоматически прогоняются через CodeRunner и Jobe командой:

```bash
docker compose exec -T moodle php /opt/python-course/check.php --run-reference
```

## Безопасность

В проекте применены следующие меры:

- база, Jobe, Ollama и AI service не публикуются наружу;
- секреты хранятся в `.env`, который исключён из Git;
- код не выполняется в Moodle или AI service;
- скрытые поля фильтруются по разрешённому списку;
- запрос анализа ограничен capability, `sesskey` и лимитом;
- HTML из ответа модели экранируется;
- полный готовый код по умолчанию запрещён;
- удаление volumes требует `CONFIRM=yes`.

## Развёртывание

На учебном сервере проект запускается после клонирования и создания `.env`. Для сервера с 4 ГБ RAM используется модель `qwen2.5-coder:0.5b`. Уже работающий Jobe на порту 9000 не конфликтует со стендом, потому что Jobe проекта не публикует порт хоста.

Для переноса на действующий Moodle института копируется только собственный плагин, запускаются AI service и Ollama, а endpoint задаётся в настройках Moodle. Подробная последовательность приведена в `docs/DEPLOYMENT.md`.
