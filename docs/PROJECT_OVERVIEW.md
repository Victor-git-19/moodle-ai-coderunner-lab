# Обзор проекта

## Какую задачу решает стенд

Проект объединяет Moodle 5.2.1, автоматическую проверку программ и локальные ИИ-подсказки. Студент решает обычный вопрос CodeRunner. После проверки он может запросить разбор именно своей попытки, не копируя условие, код и результаты тестов вручную.

CodeRunner остаётся источником истины: он выставляет результат по тестам. AI service только объясняет возможную причину ошибки и предлагает следующий шаг.

## Сервисы

- `moodle` — сайт, курсы, вопросы, попытки и собственный плагин;
- `db` — MariaDB с данными Moodle и AI-кэшем;
- `jobe` — безопасное выполнение программ для CodeRunner;
- `ai-service` — FastAPI, статический анализ и запрос к модели;
- `ollama` — локальный запуск модели;
- `ollama-model` — одноразовая проверка и загрузка модели.

Все сервисы общаются по DNS-именам Compose. Наружу открыт только порт Moodle.

## Что было готовым

Moodle, CodeRunner, behaviour-плагин, Jobe, Ollama и модель Qwen — готовые сторонние компоненты с закреплёнными версиями. Их исходники в проекте не изменяются.

## Что сделали мы

В проекте разработаны:

- простое Compose-развёртывание;
- автоматическая первая установка Moodle;
- FastAPI-сервис с безопасным `ast`-анализом;
- fallback при недоступной модели;
- Moodle-плагин `local_aicodehelper`;
- сбор данных текущей попытки CodeRunner;
- фильтрация скрытых тестов;
- серверный кэш анализа;
- проверки, smoke-тест и инструкции переноса.

## Как проверяется Python

Студент отправляет ответ в Moodle. CodeRunner формирует запуск по тестам и обращается к Jobe. Jobe запускает Python с ограничениями и возвращает stdout, stderr, ошибки и статус. CodeRunner сравнивает результат с ожидаемым и выставляет балл.

AI service никогда не выполняет пользовательский код. Это важно: синтаксическое дерево можно безопасно изучить через `ast.parse`, а фактическое выполнение остаётся только в специально предназначенном Jobe.

## Как работает AI

После проверки hook Moodle добавляет кнопку к попытке CodeRunner. По нажатию браузер отправляет только идентификаторы попытки и `sesskey` в `ajax.php`.

Сервер Moodle:

1. проверяет авторизацию, capability, sesskey и доступ к попытке;
2. сам читает вопрос, ответ и результат CodeRunner;
3. строит безопасный payload;
4. проверяет кэш текущего пользователя, попытки, слота и шага;
5. вызывает AI service по внутреннему адресу;
6. экранирует все строки ответа и возвращает готовый HTML.

AI service сначала выполняет статический анализ. Затем он просит Ollama вернуть структурированный преподавательский ответ. Если модель недоступна, используется статический fallback с тем же JSON-контрактом.

## Зачем MariaDB

MariaDB хранит всё постоянное состояние Moodle: пользователей, курсы, вопросы, попытки, настройки и таблицу AI-кэша. Поэтому обычный restart контейнеров не теряет учебные данные.

## Зачем Jobe

Jobe отделяет выполнение недоверенного кода от Moodle и AI service. CodeRunner управляет тестами, а Jobe отвечает за запуск программы и ограничения ресурсов.

## Зачем Docker

Docker фиксирует версии и связи компонентов. На другом Mac или Ubuntu-сервере не нужно вручную собирать PHP, Moodle, MariaDB, Jobe, Python и Ollama. Именованные volumes сохраняют базу, `moodledata` и модели.

Фиксированная подсеть `198.18.0.0/24` оставлена намеренно. При обычной приватной подсети защищённый HTTP-клиент Moodle 5.2.1 блокирует запрос CodeRunner к Jobe. Это проверено реальным запуском, а не только открытым портом.

## Защита скрытых тестов

`payload_builder.php` работает по принципу allowlist. Для видимого теста он берёт только столбцы, которые CodeRunner уже разрешил показать пользователю. Для скрытого теста остаются только номер, успех или провал, безопасный тип ошибки и короткое сообщение.

Не передаются:

- скрытый вход;
- ожидаемый ответ;
- test code;
- фактический скрытый вывод;
- данные шаблона;
- эталонное решение преподавателя.

Это отдельно проверяется Moodle CLI-тестом и smoke-тестом с маркерами `SECRET_*`.

## Как работает fallback

При timeout, недоступной Ollama или неверном JSON AI service не падает. Он возвращает результат статического анализа с `fallback_used: true`. Страница Moodle продолжает работать, а студент получает ограниченную, но понятную подсказку.

## Как переносить проект

На чистом Ubuntu проще перенести весь репозиторий, создать `.env`, собрать и запустить Compose.

На действующий Moodle 5.2.1 полный контейнер не нужен. Переносятся собственный `local_aicodehelper`, AI service, Ollama и настройки. CodeRunner, behaviour и Jobe добавляются только при необходимости. Сначала обязательны backup и проверка на копии. Подробности: [DEPLOYMENT.md](DEPLOYMENT.md).

## Пять файлов, которые стоит понять

1. `compose.yaml` — сервисы, их зависимости, healthcheck, сеть и volumes.
2. `moodle/entrypoint.sh` — первая установка Moodle и безопасный повторный upgrade.
3. `moodle/local_aicodehelper/classes/payload_builder.php` — чтение попытки и защита скрытых тестов.
4. `ai-service/app/main.py` и связанные модули — API, схемы, статический анализ, Ollama и fallback.
5. `scripts/smoke-test.sh` — реальный путь Moodle → CodeRunner → Jobe и Moodle → AI service.

## Пять команд, которые стоит помнить

```bash
make up
make check
```
